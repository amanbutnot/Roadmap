Swift Language Fundamentals

~14–20 days
🔤
Swift Syntax Basics
Easy 3–5 days Foundation

Swift is Kotlin's sister language. let = val, var = var, nil = null. Optionals work like Kotlin nullable types — String? can be nil. Unwrap with if let, guard let, or ! (force unwrap — dangerous). String interpolation uses \(variable) instead of ${variable}. Arrays use [1,2,3], dictionaries use ["key":"value"]. Type inference works identically to Kotlin.

val/var → let/var · null → nil · ${x} → \(x) · ?: → ?? · !! → !
Key Concepts
let / varString?if letguard let?? nil coalescingstring interpolationarraysdictionariestype inference

Enums, Structs & Switch
Easy 3–4 days Foundation

Swift enums with associated values are Kotlin sealed classes. enum State { case loading; case success(String); case failure(Error) }. The switch statement is exhaustive by default — it won't compile if you miss a case. Structs are value types — copying creates a completely independent copy. SwiftUI Views are always structs. Classes are reference types like Kotlin.

sealed class → enum with associated values · when → switch · data class → struct (mostly)
Key Concepts
enum associated valuesswitch exhaustivenessstruct (value type)class (reference type)computed propertiesmutating func

Functions, Closures & Higher-Order Functions
Medium 3–5 days Foundation

Swift functions have argument labels — func greet(for person: String) — the external label for is used at the call site. Closures use trailing syntax: when the last argument is a closure, move it outside the parentheses. This is how every SwiftUI modifier works — Button("Tap") { doSomething() }. @escaping means the closure outlives the function — used for async callbacks. filter/map/reduce work identically to Kotlin with $0 shorthand.

fun → func · lambdas → closures · it → $0 · trailing lambda → trailing closure
Key Concepts
argument labelstrailing closure@escaping$0 shorthandfilter/map/reducecompactMapgenerics basics

Protocols, Extensions & Codable
Medium 4–6 days Foundation

Protocols are interfaces but more powerful. Extensions can add protocol conformance to ANY type retroactively — even types you don't own. Codable = automatic JSON encode/decode (like @Serializable). Identifiable is required for SwiftUI Lists — needs a unique id property. Equatable and Hashable work just like Kotlin. Protocol default implementations via extensions replace abstract classes.

interface → protocol · extension fun → extension · @Serializable → Codable · data class id → Identifiable
Key Concepts
protocol conformanceextensionCodableIdentifiableEquatable / HashableJSONEncoder / JSONDecoderprotocol composition

Your First View — Text, Image, Stacks
Easy 2–3 days SwiftUI Core

A SwiftUI View is a struct conforming to the View protocol with a body computed property — like a Composable function but as a struct. VStack = Column, HStack = Row, ZStack = Box. Modifiers chain and order matters — .padding().background() ≠ .background().padding(). SF Symbols give you 5000+ free icons via Image(systemName:). Always use system colours — dark mode works free.

@Composable fun → struct : View · Modifier chain → .modifier().modifier() · painterResource → Image(systemName:)

Layout System — frame, GeometryReader, LazyStacks
Medium 4–5 days SwiftUI Core

SwiftUI layout is a 3-step negotiation: parent proposes size → child picks its own size → parent positions child. Text shrinks to content, Color expands to fill. Use .frame() to constrain sizes. GeometryReader gives you the parent's size at runtime. LazyVStack/LazyHStack = lazy loading (like LazyColumn). overlay and background stack views without ZStack. Grid layouts use LazyVGrid / LazyHGrid.

LazyColumn → LazyVStack in ScrollView · BoxWithConstraints → GeometryReader · fillMaxSize → .frame(maxWidth:.infinity)
Key Concepts

@State, @Binding — Local State
Medium 3–5 days State Management

@State is local, private state that lives inside one View — like remember { mutableStateOf() } in Compose. When it changes, SwiftUI re-evaluates the body. @Binding is a two-way connection to a parent's @State — like passing a lambda to update parent state in Compose, but cleaner. Pass a binding with $variableName. Always mark @State as private.

remember { mutableStateOf() } → @State · hoisted state callback → @Binding · state hoisting concept is identical

List, ForEach & Dynamic Content
Easy 2–3 days SwiftUI Core

List is the iOS equivalent of LazyColumn — it is lazy by default and styled. Items must conform to Identifiable. ForEach is the loop construct inside Views. Swipe-to-delete with .onDelete. Reorder with .onMove. Section headers with Section. Customise the list style with .listStyle().

Navigation — NavigationStack, Sheets, Tabs
Medium 4–6 days Navigation

NavigationStack is the modern navigation (iOS 16+) — it uses a path array to represent the back stack, like Compose Navigation's NavController. NavigationLink pushes a new view. Deep linking works by populating the path. .sheet() presents a modal. .fullScreenCover() covers the whole screen. TabView creates a bottom tab bar. .navigationTitle() sets the header title.

@Observable, @ObservableObject — ViewModels
Medium 4–6 days State Management Critical

For shared, business-logic state use a class marked @Observable (iOS 17+, modern) or ObservableObject with @Published (iOS 14+). This is SwiftUI's ViewModel. Use @StateObject to own it, @ObservedObject to receive it. @EnvironmentObject injects it globally — like Compose's CompositionLocalProvider. Never create an @ObservedObject inside the view body — it will reset on recomposition.

ViewModel + StateFlow → @Observable class · collectAsState() → @ObservedObject · CompositionLocal → @EnvironmentObject

async/await & Concurrency
Medium 3–5 days Concurrency

Swift's async/await is conceptually identical to Kotlin coroutines — suspend functions become async func. Task { } launches a coroutine. await suspends. async let = parallel execution (like async {} in Kotlin). @MainActor ensures UI updates on main thread — like withContext(Dispatchers.Main). SwiftUI's .task { } modifier launches a Task tied to the view's lifetime — auto-cancelled when view disappears.

suspend fun → async func · coroutineScope → TaskGroup · Dispatchers.Main → @MainActor · LaunchedEffect → .task { }

Networking & Data Persistence

~14–20 days
🌐
Networking — URLSession, REST APIs
Medium 4–6 days Networking

iOS networking uses URLSession — no Retrofit, no Ktor. Make a request with URLSession.shared.data(from: url) which is async. Decode the response with JSONDecoder into your Codable model. Error handling uses Swift's try/catch — async functions throw errors. Build a clean network layer with a NetworkService class. Use URLRequest for POST/headers/auth. Handle 401, 404, 500 with typed errors.

Retrofit/Ktor → URLSession · @Serializable → Codable · Result<T,E> → throws / try-catch

SwiftData & Core Data — Local Persistence
Medium 4–6 days Persistence Modern

SwiftData (iOS 17+) is the modern ORM — mark your model with @Model and it handles persistence automatically. Like Room but with zero boilerplate. @Query in your view fetches and updates live. Use modelContext.insert() and modelContext.delete(). For iOS 16 support, learn Core Data basics — it is the predecessor. UserDefaults = SharedPreferences for simple key-value storage.

Room @Entity → @Model · DAO queries → @Query · SharedPreferences → UserDefaults

Keychain & Secure Storage
Medium 2–3 days Security

The Keychain is iOS's secure credential store — encrypted by the OS, survives app reinstalls, locked when device is locked. Use it for tokens, passwords, and sensitive keys. Never store these in UserDefaults. The raw Keychain API is verbose C-style — use a wrapper library like KeychainSwift or write a thin wrapper yourself. Keychain items have access control — kSecAttrAccessibleWhenUnlocked is the safest default.

EncryptedSharedPreferences → Keychain · same threat model, different API

Animations — Implicit, Explicit, Transitions
Medium 4–6 days UI Polish

Implicit animations: add .animation(.easeInOut, value: someState) — SwiftUI animates any change to that value automatically. Explicit animations: withAnimation { state = newValue } wraps state changes. Transition controls how views appear/disappear — .slide, .scale, .opacity. matchedGeometryEffect creates hero animations between views. Spring animations feel native and are preferred over linear.

Gestures — Tap, Drag, Long Press, Pinch
Medium 3–4 days Interaction

SwiftUI gestures attach to any view via modifiers. .onTapGesture for taps. DragGesture gives you translation and velocity for drag-to-dismiss cards. MagnificationGesture for pinch-to-zoom. LongPressGesture for context menus. Combine gestures with .simultaneously, .sequenced, or .exclusively. Use @GestureState to track gesture progress without triggering full redraws.

Canvas & Custom Drawing
Medium 3–4 days Custom UI

SwiftUI's Canvas view lets you draw paths, shapes, and images using a GraphicsContext. Think Android's Canvas with DrawScope. Use Path to define shapes with bezier curves, arcs, lines. Built-in shapes: Circle, RoundedRectangle, Capsule. Custom shapes by conforming to the Shape protocol. TimelineView drives animated canvas at 60fps.

Always start with when-in-use. MapKit renders maps with Map view (SwiftUI native). Add pins with Marker and Annotation. Geocoding with CLGeocoder. NFC & Bluetooth
Hard 4–6 days Hardware

NFC reading uses CoreNFC — NFCTagReaderSession for reading tags. iOS can read NFC but not emulate (no HCE like Android). Add NFCReaderUsageDescription to Info.plist. Bluetooth uses CoreBluetooth — CBCentralManager scans and connects, CBPeripheralManager advertises. BLE (Bluetooth Low Energy) is the default on modern iOS. State restoration lets BLE connections survive app termination.

iOS has no HCE — NFC write not possible · BluetoothAdapter → CBCentralManager · BLE API is very similar conceptually

Push Notifications — APNs & Firebase
Hard 4–6 days Notifications

Apple Push Notification Service (APNs) is iOS's push system — the equivalent of FCM but lower level. Most apps use Firebase Cloud Messaging which wraps APNs. You MUST have an Apple Developer account ($99/year). Flow: register device → get APNs token → send token to your server → server sends push via APNs/FCM. UNNotificationServiceExtension lets you modify push payloads before display — used for rich notifications with images. UNNotificationContentExtension creates custom notification UI.

FCM is the same — it wraps APNs just as it wraps FCM on Android. Token registration flow is identical.

Background Execution

~10–14 days
⚙️
Background Tasks — BGTaskScheduler
Hard 4–6 days Background Critical

iOS is extremely aggressive about killing background apps — this is the biggest difference from Android. BGTaskScheduler is the modern API (iOS 13+) — equivalent to WorkManager. Two types: BGAppRefreshTask (short, 30 seconds max — for refreshing content) and BGProcessingTask (longer, minutes — for heavy sync). iOS decides WHEN to run them based on battery, usage patterns, and connectivity. You cannot force them to run at an exact time. Register identifiers in Info.plist. Always call task.setTaskCompleted(success:) or iOS penalises future scheduling.

WorkManager → BGTaskScheduler · PeriodicWorkRequest → BGAppRefreshTask · The key difference: iOS controls WHEN it runs, you don't.
Key Concepts
BGTaskSchedulerBGAppRefreshTaskBGProcessingTasktask identifiersetTaskCompletedexpiration handler
🌙
Background Modes — Audio, Location, VOIP, Fetch
Hard 3–5 days Background

Some apps are allowed to run in background indefinitely with special entitlements declared in capabilities. Audio: music/podcast players stay alive playing audio. Location: navigation apps receive continuous location updates. VoIP: calls apps stay alive for incoming calls. Background fetch: periodic content refresh. Remote notifications: silent push wakes the app for 30 seconds. You must justify each capability to Apple during review — adding them without a genuine reason causes rejection. Foreground services don't exist on iOS.

Foreground Service does NOT exist on iOS. Background Modes are the closest — but far more restricted and require Apple approval.

Background Downloads & Uploads
Medium 2–3 days Background

URLSession with a background configuration (URLSessionConfiguration.background(withIdentifier:)) continues downloads even if your app is killed. iOS handles the download and wakes your app when done via application(_:handleEventsForBackgroundURLSession:). Use this for large file downloads. Progress updates come via delegates. The background identifier must match between sessions for reconnection after app relaunch.

DownloadManager → URLSession background configuration · Same mental model: OS handles the download, notifies you when done.

WidgetKit Basics — Timeline & Entry
Medium 4–5 days Widgets iOS Unique

iOS widgets use WidgetKit — completely different from Android's RemoteViews/Glance. The key concept: widgets are not live views. You provide a Timeline — a sequence of TimelineEntry snapshots with associated dates. iOS renders each snapshot at the right time, like a flipbook. Your widget view is SwiftUI — much nicer than RemoteViews. A TimelineProvider has three methods: placeholder, getSnapshot, getTimeline. Reload policy controls when iOS asks for a new timeline.

Completely different from Android widgets. No RemoteViews. Pure SwiftUI views. Timeline = pre-rendered snapshots, not live updates.

Widget Configuration & Intents

Static widgets: no configuration. Configurable widgets use AppIntentConfiguration (iOS 17+, modern) or IntentConfiguration (iOS 14+). The user long-presses the widget to configure it — like Android's config Activity but built into iOS. AppIntent defines the parameters. Widget taps use Link or widgetURL to deep link into the app. WidgetCenter.shared.reloadAllTimelines() forces a refresh from inside the app.

Deep Links & Universal Links
Medium 3–4 days Platform

Two systems: URL Schemes (myapp://) — simple but any app can claim the same scheme. Universal Links (https://yourdomain.com/path) — verified ownership via apple-app-site-association file hosted on your server. Universal Links are the modern standard — they fall back to a website if the app isn't installed. Handle in SwiftUI with .onOpenURL modifier. Performance & Architecture

Performance — Instruments, Memory, Redraws
Hard 4–6 days Performance

Xcode Instruments is your profiler — equivalent to Android Studio's profiler. Use the Time Profiler for CPU, Allocations for memory leaks, SwiftUI instrument for excessive redraws. SwiftUI redraws the body of any view whose observed state changes — use equatable() to prevent unnecessary redraws. Break large views into smaller structs. Avoid creating objects in body. @StateObject vs @ObservedObject ownership matters for memory.

Android Studio Profiler → Instruments · recomposition debugging → SwiftUI Instrument · remember { } leaks → @StateObject lifetime

Architecture — MVVM, TCA, Clean
Medium 4–5 days Architecture

MVVM is the default SwiftUI pattern — @Observable class = ViewModel, View is the V, your model is the M. Identical to Android MVVM with StateFlow. TCA (The Composable Architecture) by Point-Free is popular for testability — State + Action + Reducer + Effect, like MVI/Redux. Clean Architecture layers (Presentation → Domain → Data) map directly from your Android knowledge. For solo apps, start with MVVM. Add TCA when you need strict testability.

MVI/Redux → TCA · ViewModel + StateFlow → @Observable class · Repository pattern is identical

Signing, Certificates & App Store Connect
Hard 3–5 days Deployment One-time Pain

The most confusing part of iOS for Android developers. You need a $99/year Apple Developer account. Key concepts: Bundle ID (like application ID), Provisioning Profile (ties your app + device + certificate — no Android equivalent), Signing Certificate (proves it's you). App review takes 24–48 hours on average. TestFlight is Apple's beta distribution (like Firebase App Distribution).
