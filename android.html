🧠 Why This Module

This is your most critical weak spot. If you don't understand when to use StateFlow vs SharedFlow, every screen you build will have subtle bugs — duplicate events, missed updates, or state that resets unexpectedly. This module fixes that mental model permanently.

📱 Mini App to Build

Counter + Snackbar + Navigation Trigger App
A single screen with a number counter, a button that fires a one-shot Snackbar message, and a button that navigates to a second screen. Simple on the surface — but it forces you to decide the exact right Flow type for every piece of state.

🎯 Core Focus

StateFlow vs SharedFlow · UI state vs events · recomposition control

📖 Deep Explanation

StateFlow is for state — things that have a current value the UI needs to display. The counter number is state. Whether the button is enabled is state. SharedFlow is for events — things that happen once and must be consumed exactly once. A Snackbar message is an event. A navigation command is an event. The critical mistake most devs make is using StateFlow for events. When the screen rotates, the collector resubscribes and replays the last value — so your Snackbar fires again, your navigation triggers twice, your user gets confused. You must feel this bug yourself, then fix it. That is the whole point of this module.

Collect StateFlow in Composable and display the number
Add a SharedFlow<String> for Snackbar events (replay = 0)
Wire a button to emit a Snackbar event via SharedFlow
Add a SharedFlow<Unit> for navigation trigger to Screen 2
Handle both SharedFlows in LaunchedEffect with correct keys
Write inline comments explaining WHY each Flow type was chosen

⚠️ Constraints (Rules You Must Not Break)

· Zero LiveData anywhere in the project
· No mutable state held directly in a Composable (no remember { mutableStateOf } for business data)
· Events must be consumed exactly once — never replayed on recomposition
· No GlobalScope usage

🧠 Why This Module

Coroutines are the engine of every Android app you will build. If you misuse withContext you will block the UI thread and cause ANRs. If you misuse async/await you will run things sequentially when they should be parallel. If you don't handle cancellation, you will leak work and crash on destroyed ViewModels. This module makes all of that muscle memory.

📱 Mini App to Build

Parallel Task Runner with Cancel, Timeout, and Error UI
An app that simulates fetching data from 3 independent sources at the same time (fake delays with kotlinx.coroutines.delay). Shows a combined result only when all complete, or a partial failure state if one fails. Has a Cancel button that stops everything mid-flight.

🎯 Core Focus

launch vs async · withContext · Dispatchers · structured concurrency · cancellation

📖 Deep Explanation

launch fires a coroutine and forgets about it — use it for side effects. async fires a coroutine and gives you a Deferred you can await() — use it when you need a result. withContext switches the coroutine to a different dispatcher for the duration of a block, then switches back. Use Dispatchers.IO for file/network work, Dispatchers.Default for CPU work. Never do either on Dispatchers.Main. Structured concurrency means: if the parent scope is cancelled, all children are cancelled too. This is why you should NEVER use GlobalScope — it has no parent, so cancellation doesn't work.

🧠 Why This Module

Configuration changes and lifecycle events break more apps than any other single cause. Timers reset, forms clear, network calls duplicate. If you don't understand the difference between viewModelScope and lifecycleScope — and when each one is alive — your apps will behave inconsistently in ways that are nearly impossible to debug.

📱 Mini App to Build

Stopwatch That Survives Rotation and Pauses on Background
A stopwatch with Start, Pause, and Reset. Survives screen rotation without resetting. Automatically pauses when the app goes to background and resumes when it returns. Logs every lifecycle event to an on-screen scrollable list so you can SEE the lifecycle happening.

🎯 Core Focus

ViewModel lifecycle · viewModelScope · lifecycleScope · repeatOnLifecycle

📖 Deep Explanation

ViewModel survives configuration changes — the Activity/Fragment is destroyed and recreated, but the ViewModel is not. This is why you store your timer logic there. viewModelScope lives as long as the ViewModel — it's cancelled when the user leaves the screen permanently. lifecycleScope lives as long as the Activity/Fragment — it's cancelled on every configuration change. repeatOnLifecycle(Lifecycle.State.STARTED) is the correct way to collect flows from the UI — it automatically stops collecting when the app goes to background and restarts when it returns. This is why your stopwatch should auto-pause on background without any manual code.

🧠 Why This Module

Home screen widgets are the most visible surface of your app — they live on the launcher 24/7 and users judge your app by them. Most Android devs avoid widgets because the RemoteViews system is confusing and the lifecycle is nothing like activities. This module kills that confusion permanently. 📱 Mini App to Build

Static Info Widget — Date, Time, and a Counter
A widget that shows the current date, a live clock that ticks every minute, and a tap-to-increment counter that persists across reboots. Forces you to deal with PendingIntent, WorkManager scheduling, and SharedPreferences-based state — all without a running Activity.

🎯 Core Focus

AppWidgetProvider · RemoteViews · PendingIntent · widget lifecycle · update scheduling

📖 Deep Explanation

Widgets run in the launcher's process — not yours. You cannot use Compose, View binding, or any direct UI reference. RemoteViews is a serialisable description of a layout that is inflated by the launcher. You send it updates via AppWidgetManager.updateAppWidget(). Widget updates are triggered by: the system (every UPDATEPERIODMILLIS), a BroadcastReceiver you fire manually, or a WorkManager job. UPDATEPERIODMILLIS has a minimum of 30 minutes enforced by the OS — for sub-30-minute updates you MUST use WorkManager or AlarmManager. Tap actions use PendingIntent — you cannot use click listeners. Every tap must map to a broadcast, an activity launch, or a service start. Persistence: widgets survive process death, so all state must live in SharedPreferences or a database, never in memory.

Use RemoteViews to set text for date and time fields
Wire a tap on the counter TextView to a PendingIntent that fires a custom broadcast
Handle the broadcast in onReceive() — increment counter in SharedPreferences, call updateAppWidget()
Schedule a WorkManager PeriodicWorkRequest to refresh the clock every 15 minutes
Implement onEnabled() and onDisabled() to start/cancel the WorkManager job
Implement onDeleted() to clean up SharedPreferences state for that widget ID
Add the widget to the home screen and verify counter persists after device reboot

⚠️ Constraints (Rules You Must Not Break)

· Zero Jetpack Compose in the widget layout — RemoteViews only
· No updating the widget from an Activity — all updates must go through AppWidgetProvider or WorkManager
· Counter state must survive process death and reboot (SharedPreferences with the appWidgetId as key)
· UPDATEPERIODMILLIS must not be set below 1800000 — use WorkManager for frequent updates

🧠 Why This Module

Glance is Google's Compose-based widget API — it lets you write widget UIs with a Compose-like DSL instead of raw RemoteViews. It is the modern way to build widgets and is already used in production apps. The catch: Glance is NOT Jetpack Compose. The API looks identical but has completely different constraints, a different state system, and different recomposition rules. This module makes you understand exactly where they diverge so you never confuse the two.

📱 Mini App to Build

Task List Widget with Glance
A widget that shows your top 3 pending tasks from a Room database. Tapping a task marks it done. A "+" button launches the app to add a new task. The widget list updates live as the database changes via a Flow collector running in GlanceStateDefinition.

🎯 Core Focus

GlanceAppWidget · GlanceStateDefinition · ActionCallback · Glance vs Compose differences · Flow-to-widget updates

📖 Deep Explanation

Glance wraps RemoteViews under the hood — it compiles your Glance composables into RemoteViews at runtime. This means: not all Compose Modifiers exist in Glance, LazyColumn does not exist (use Column with a loop), Canvas does not exist, and animations do not work. State in Glance uses GlanceStateDefinition — typically DataStore. You cannot use ViewModel inside a Glance widget. To update widget state when your database changes, collect a Flow in a coroutine launched from GlanceAppWidget.update() or from a Worker. ActionCallbacks handle taps — they run in a coroutine and can call suspend functions, update DataStore, then call GlanceAppWidget.update() to trigger a recomposition. The key mental model: Glance recomposition is triggered manually by you calling update(), not automatically like Compose.

Display a list of 3 task strings from DataStore in the widget UI
Create an ActionCallback for marking a task complete — updates DataStore and calls update()
Create an ActionCallback for the "+" button — uses actionStartActivity<MainActivity>()
In the app, whenever the Room task list changes, push the top 3 tasks into DataStore
Register the widget in AndroidManifest with GlanceAppWidgetReceiver

⚠️ Constraints (Rules You Must Not Break)

· No ViewModel usage inside GlanceAppWidget — state via DataStore only
· No LazyColumn — Glance does not support it, use a regular Column with forEach
· update() must be called after every DataStore write that should be reflected in the widget
· Widget must work correctly with zero tasks, one task, and three tasks

🎯 Core Focus

AppWidgetConfigure activity · EXTRAAPPWIDGETID · per-instance SharedPreferences · reconfiguration

📖 Deep Explanation

When a user adds a configurable widget, the launcher sends ACTIONAPPWIDGETCONFIGURE to your configuration Activity — NOT to AppWidgetProvider. Your Activity receives the appWidgetId via EXTRAAPPWIDGETID. You MUST set a result with RESULTOK and the same appWidgetId before finishing, or the widget is immediately removed by the launcher. After the user confirms config, you: 1) save the config to SharedPreferences keyed by appWidgetId, 2) call AppWidgetManager.updateAppWidget() with the initial RemoteViews, 3) set Activity result and finish. onUpdate() will then be called on future updates — read the saved config using the appWidgetId to reconstruct the correct RemoteViews. If the user sets RESULTCANCELED or presses back, the widget is removed — handle this correctly.

📱 Mini App to Build

Live Crypto Price Widget
A widget showing the live price of one configurable coin (BTC, ETH, SOL). Refreshes every 15 minutes via WorkManager. Shows last-updated timestamp. Colour-codes the price green/red based on 24h change. On tap, opens the app to the full price chart. Handles network failure gracefully — shows last known price with a "stale" indicator.

🎯 Core Focus

WorkManager + widget · CoroutineWorker · stale data UX · efficient RemoteViews updates · battery-safe refresh

📖 Deep Explanation

The wrong approach: updating the widget in onUpdate() by making a network call directly in the AppWidgetProvider. AppWidgetProvider callbacks run on the main thread with a 10-second timeout — any async work that outlives the broadcast will be killed. The correct approach: onUpdate() enqueues a WorkManager job. The Worker makes the network call in a coroutine, writes the result to SharedPreferences, then calls AppWidgetManager.updateAppWidget() with fresh RemoteViews. WorkManager's PeriodicWorkRequest handles scheduling. Use setForeground() if the work takes more than a few seconds (rare for a single API call, but required for robustness). For battery: use Constraints.Builder().setRequiredNetworkType(NetworkType.CONNECTED) so the Worker only runs when online — never retry endlessly on airplane mode.

In onUpdate(), enqueue a PeriodicWorkRequest (15 min interval) with KEEP existing work policy
Add a "Refresh now" tap that enqueues a one-time WorkRequest (REPLACE policy)

· Zero network calls in AppWidgetProvider callbacks — all network work goes through WorkManager
· Worker must use NetworkType.CONNECTED constraint — no blind retries on no-connection
· Use KEEP for periodic work and REPLACE for manual refresh — never APPEND (creates duplicate chains)
· Stale data must be visually distinguished from fresh data — never silently show old prices as current
