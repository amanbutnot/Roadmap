PHASE 1

🧱 Phase 1 — Core Language

Goal: Master Go's type system, variables, control flow, and fundamental syntax.

Module 1.1 — Variables & Types

Declare variables with var: var name string = "Go"
Use short declaration: name := "Go" — understand when you can and cannot use it
Know all basic types: int, int8/16/32/64, uint, float32, float64, string, bool, byte, rune
Understand zero values: uninitialized int = 0, string = "", bool = false
Declare multiple variables in a block with var ( ... )
Constants with const and typed/untyped constants
Understand iota for auto-incrementing constants
const (
    Sunday = iota   // 0
    Monday          // 1
    Tuesday         // 2
)

Module 1.2 — Strings & Runes

Strings in Go are immutable byte slices — understand the difference from C strings
String concatenation with + and efficient building with strings.Builder
Use the strings package: Contains, HasPrefix, Split, TrimSpace, ToLower, Replace
Understand UTF-8: a rune is a Unicode code point (int32), not a byte
Iterate over a string with for i, r := range s — r is a rune, not a byte
Convert string ↔ []byte and string ↔ []rune
Format strings with fmt.Sprintf and verbs: %s, %d, %f, %v, %T, %+v

Module 1.3 — Control Flow

if / else if / else — no parentheses needed around condition
Init statement in if: if err := doThing(); err != nil { ... }
switch statement — no break needed, cases don't fall through by default
Switch with no condition (replaces long if-else chains)
for loop — Go's only loop keyword. Know all three forms:
• C-style: for i := 0; i < 10; i++ {}
• While-style: for condition {}
• Infinite: for {} with break
range over slices, maps, strings, channels
break, continue, and labeled breaks for nested loops
goto — know it exists, understand why you rarely use it

Module 1.4 — Pointers

A pointer holds the memory address of a value: var p *int
Get address of variable: p = &x
Dereference pointer: *p = 42 — changes the original value
Go has NO pointer arithmetic — safer than C
The new(T) function allocates zeroed memory and returns a pointer
Know when to use pointer receivers vs value receivers (covered deeply in Phase 3)
Nil pointers — dereferencing a nil pointer causes a panic
x := 42
p := &x
*p = 100
fmt.Println(x) // 100

Module 1.5 — Type Conversions & Type Aliases

Go does NOT do implicit type conversion — always explicit: float64(i)
Convert between numeric types, string/int (strconv.Itoa, strconv.Atoi)
Define a type alias: type Celsius float64 — it's a NEW type, not the same as float64
Understand type definitions vs type aliases (type A = B vs type A B)

Module 1.6 — Scope & Shadowing

Go is lexically scoped — variables are only visible in the block they are declared
Package-level vs function-level vs block-level scope
Variable shadowing: inner := can shadow outer variable — common source of bugs
The blank identifier _ — discard values you don't need

Module 1.7 — Defer, Panic & Recover

defer schedules a function call to run when the surrounding function returns
Multiple defers execute in LIFO order (last in, first out)
Classic use: defer file.Close() right after opening a file
panic() stops normal execution and unwinds the stack
recover() inside a deferred function can catch a panic and prevent crash
Rule of thumb: use panic only for truly unrecoverable programmer errors; use error for normal failures
Phase 1 Break Test: Given a prompt, write a program with variables, constants, loops, switch, string formatting, and a working defer — from scratch, no docs.
PHASE 2

📦 Phase 2 — Data Structures & Collections

Goal: Master arrays, slices, maps, and structs — Go's core data containers.

Module 2.1 — Arrays

Declare: var arr [5]int — size is part of the type in Go
Arrays are value types — assignment copies the entire array
Array literals: arr := [3]string{"a", "b", "c"}
Use [...]int{1, 2, 3} to let Go count the size
In practice, slices are used far more than arrays — understand why

Module 2.2 — Slices (Critical)

A slice is a descriptor: pointer + length + capacity — it does NOT own memory, the array underneath does
Create with literal: s := []int{1, 2, 3}
Create with make: s := make([]int, len, cap)
Slice a slice: s[1:3] — shares underlying array
Append: s = append(s, 4, 5) — may or may not allocate a new array
Understand when append causes a new allocation (capacity exceeded) vs in-place growth
Copy: copy(dst, src) — copies min(len(dst), len(src)) elements
Nil slice vs empty slice — both have len 0, but nil slice is == nil
2D slices: [][]int — create and iterate

Module 2.3 — Maps

Declare: m := map[string]int{} or make(map[string]int)
Set: m["key"] = 42, Get: v := m["key"]
Check existence: v, ok := m["key"] — always use two-value form
Delete: delete(m, "key")
Iterate: for k, v := range m — order is NOT guaranteed
Maps are reference types — passing a map to a function modifies the original
Maps are NOT safe for concurrent reads and writes — use sync.Map or a mutex

Module 2.4 — Structs

Declare a struct type: type Person struct { Name string; Age int }
Create struct literals: named fields vs positional (prefer named)
Access and set fields with dot notation
Structs are value types — assignment copies the struct
Anonymous structs: p := struct{ X, Y int }{X: 1, Y: 2}
Embedded structs (Go's form of composition): type Employee struct { Person; Salary float64 }
Struct tags for JSON: Name string \`json:"name"\`
Comparing structs — two structs are equal if all fields are equal and all fields are comparable

Module 2.5 — Generics Basics (Go 1.18+)

Understand why generics exist: write one function that works for multiple types
Type parameters: func Min[T constraints.Ordered](a, b T) T
The constraints package: any, comparable, constraints.Ordered
Generic structs: type Stack[T any] struct { items []T }
When NOT to use generics — don't over-engineer simple code
Phase 2 Break Test: Implement a generic stack, a word-frequency counter (map), and a function that filters a slice — all from memory, with tests.
PHASE 3

🔧 Phase 3 — Functions, Methods & Interfaces

Goal: Master Go's function system and its elegant, implicit interface model.

Module 3.1 — Functions

Basic function: parameters, return type, return value
Multiple return values: func divide(a, b float64) (float64, error)
Named return values and naked returns (use sparingly)
Variadic functions: func sum(nums ...int) int
Functions as first-class values: assign to variable, pass as argument, return from function
Anonymous functions / function literals: func(x int) int { return x * 2 }(5)
Closures — a function that captures variables from its enclosing scope
func counter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}

Module 3.2 — Methods

A method is a function with a receiver: func (p Person) Greet() string
Value receiver (func (p Person)) — works on a copy of the struct
Pointer receiver (func (p *Person)) — modifies the original; use for mutation or large structs
Consistency rule: if any method uses a pointer receiver, all methods should
You can define methods on any named type, not just structs: type Celsius float64; func (c Celsius) ToF() float64
Method expressions and method values

Module 3.3 — Interfaces (Core Go Concept)

An interface is a set of method signatures — implicit, no implements keyword
A type "implements" an interface by having all required methods — automatically
Define an interface: type Stringer interface { String() string }
The fmt.Stringer interface — implement it on your structs for custom printing
Interface variables hold a (value, type) pair internally
The empty interface interface{} (or any) accepts any value — use with care
Type assertion: v, ok := i.(string) — safely extract concrete type from interface
Type switch: switch v := i.(type) { case string: ... case int: ... }
Interface composition: type ReadWriter interface { Reader; Writer }
Design principle: accept interfaces, return structs
type Shape interface {
    Area() float64
    Perimeter() float64
}

type Circle struct { Radius float64 }
func (c Circle) Area() float64      { return 3.14159 * c.Radius * c.Radius }
func (c Circle) Perimeter() float64 { return 2 * 3.14159 * c.Radius }

Module 3.4 — Key Stdlib Interfaces

io.Reader — the most important interface in Go: Read(p []byte) (n int, err error)
io.Writer — Write(p []byte) (n int, err error)
io.Closer, io.ReadWriter, io.ReadWriteCloser
fmt.Stringer, fmt.GoStringer, error interface
sort.Interface: Len(), Less(i, j int) bool, Swap(i, j int)
Implement sort.Interface on a custom slice and sort it with sort.Sort

Module 3.5 — Functional Patterns

Implement Map, Filter, and Reduce as generic functions
Function options pattern for flexible constructors
Middleware pattern using function wrapping
The slices package (Go 1.21+): slices.Contains, slices.SortFunc
Phase 3 Break Test: Define a Shape interface, implement it on 3 types, write a function accepting []Shape and computing total area — with a type switch. No docs.
PHASE 4

🔴 Phase 4 — Error Handling & Packages

Goal: Handle errors the Go way — explicit, propagated, and meaningful.

Module 4.1 — Error Handling

The error type is just an interface: type error interface { Error() string }
Always check errors: if err != nil { return err } — the Go way
Create errors: errors.New("something failed")
Formatted errors: fmt.Errorf("loading user %d: %w", id, err)
The %w verb wraps an error — use errors.Unwrap, errors.Is, errors.As
errors.Is(err, target) — checks if any error in the chain equals target
errors.As(err, &target) — extracts a specific error type from the chain
Define custom error types: type NotFoundError struct { ID int } with Error() string method
Sentinel errors: package-level var ErrNotFound = errors.New("not found")
type ValidationError struct {
    Field   string
    Message string
}
func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation error on %s: %s", e.Field, e.Message)
}

Module 4.2 — Package Design

One package per directory — the directory name is the import path segment
Package naming conventions: short, lowercase, no underscores
internal/ packages — only importable by the parent module
Circular imports are forbidden — design packages to be layered
The init() function — runs once per package, before main
Blank import _ "pkg" — imports for side effects (e.g. registering SQL drivers)

Module 4.3 — Essential Stdlib Packages

fmt — formatted I/O, Println, Printf, Sprintf, Fprintf, Errorf, Sscanf
os — file system, environment variables, os.Exit, os.Args
io — io.ReadAll, io.Copy, io.Pipe, io.TeeReader
bufio — buffered reader/writer, bufio.Scanner for line-by-line reading
strings and bytes — string manipulation
strconv — string ↔ int/float conversions
math and math/rand — basic math operations
time — time.Now, time.Duration, formatting, time.Sleep, time.After
regexp — compile and use regular expressions
sort — sort slices with sort.Slice and sort.SliceStable

Module 4.4 — Logging

Standard log package: log.Println, log.Fatal, log.Printf
log/slog (Go 1.21+) — structured logging with levels: Debug, Info, Warn, Error
Configure slog with JSON handler for production output
Third-party: zerolog or zap for high-performance logging
Phase 4 Break Test: Write a package with custom error types, sentinel errors, error wrapping, and a package-level logger. Verify errors.Is and errors.As work correctly via tests.
PHASE 5 — ⭐ Most Important

⚡ Phase 5 — Concurrency (Go's Superpower)

Goal: Understand goroutines, channels, and synchronization — the reason most people choose Go.

Module 5.1 — Goroutines

A goroutine is a lightweight thread managed by the Go runtime — NOT an OS thread
Launch with go functionName() or go func() { ... }()
Go can run thousands of goroutines concurrently on a small number of OS threads
The main goroutine exiting kills all other goroutines — use sync.WaitGroup to wait
Goroutines start with a small stack (~2KB) that grows automatically
GOMAXPROCS — controls how many OS threads run goroutines in parallel
var wg sync.WaitGroup
for i := 0; i < 5; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        fmt.Println("worker", id)
    }(i)
}
wg.Wait()

Module 5.2 — Channels

Channels are typed pipes for communicating between goroutines
Create: ch := make(chan int) (unbuffered) or make(chan int, 10) (buffered)
Send: ch <- value, Receive: value := <-ch
Unbuffered channel: send blocks until someone receives — synchronization point
Buffered channel: send only blocks when buffer is full
Close a channel: close(ch) — signals no more values will be sent
Range over channel: for v := range ch — exits when channel is closed
Check if channel is closed: v, ok := <-ch — ok is false when closed
Directional channels: chan<- int (send only), <-chan int (receive only)

Module 5.3 — Select

select lets a goroutine wait on multiple channel operations simultaneously
If multiple cases are ready, one is chosen at random
default case in select — runs if no other case is ready (non-blocking)
Timeout pattern: case <-time.After(5 * time.Second):
Cancellation pattern with a done channel
select {
case msg := <-ch1:
    fmt.Println("received from ch1:", msg)
case ch2 <- 42:
    fmt.Println("sent to ch2")
case <-time.After(time.Second):
    fmt.Println("timeout")
}

Module 5.4 — sync Package

sync.WaitGroup — wait for a group of goroutines to finish
sync.Mutex — mutual exclusion lock: mu.Lock() / mu.Unlock()
sync.RWMutex — allows multiple concurrent readers, one writer
sync.Once — ensures code runs exactly once (singleton initialization)
sync.Pool — reuse temporary objects to reduce GC pressure
sync.Map — concurrent-safe map for specific use cases

Module 5.5 — Context

The context package carries deadlines, cancellations, and request-scoped values
context.Background() — root context, never cancelled
context.WithCancel(parent) — returns ctx + cancel function; call cancel() to cancel
context.WithTimeout(parent, duration) — auto-cancels after duration
context.WithDeadline(parent, time) — cancels at absolute time
Pass context as first argument to functions: func DoWork(ctx context.Context, ...)
Check for cancellation inside loops: select { case <-ctx.Done(): return ctx.Err() }
context.WithValue — store request-scoped values (use typed keys to avoid collisions)

Module 5.6 — Atomic Operations & Race Detector

sync/atomic — low-level atomic operations for int64, uint32, pointers without mutex overhead
Run the race detector: go run -race main.go — catches data races at runtime
Understand data races: two goroutines access same memory, at least one writes, no sync
Golden rule: Do not communicate by sharing memory; share memory by communicating.
Phase 5 Break Test: Build a worker pool — N goroutines process jobs from a buffered channel. Support graceful shutdown via context cancellation. Run with -race, zero races.

Module 6.1 — File Operations

Open a file: os.Open(name) (read-only) or os.OpenFile(name, flags, perm)
Always defer f.Close() after opening — check the error on Close too
Read entire file: os.ReadFile(name)
Write entire file: os.WriteFile(name, data, 0644)
Read line by line with bufio.Scanner
Write buffered with bufio.Writer — always Flush() at end
Work with temp files: os.CreateTemp, os.MkdirTemp
Walk a directory tree: filepath.Walk or fs.WalkDir

Module 6.2 — JSON Encoding/Decoding

Marshal struct to JSON: json.Marshal(v) returns ([]byte, error)
Unmarshal JSON to struct: json.Unmarshal(data, &v)
Struct tags control JSON keys: json:"name,omitempty"
Stream JSON with json.NewEncoder(w).Encode(v) and json.NewDecoder(r).Decode(&v)
Handle unknown JSON structure with map[string]interface{} or json.RawMessage
Custom marshaling: implement json.Marshaler / json.Unmarshaler interfaces
type User struct {
    ID       int    \`json:"id"\`
    Name     string \`json:"name"\`
    Password string \`json:"-"\`           // never serialized
    Email    string \`json:"email,omitempty"\`
}

Module 6.3 — net/http Basics

Make an HTTP GET: http.Get(url) — always close resp.Body
Make a POST with JSON body using http.NewRequest and http.Client.Do
Set custom headers and timeouts on http.Client
Read response body with io.ReadAll(resp.Body)
Never use the default http.Client in production — always set Timeout

Module 6.4 — Templates

text/template — render text output from a template and data struct
html/template — HTML-aware, auto-escapes values for XSS prevention
Template actions: {{.Field}}, {{if}}, {{range}}, {{define}}, {{template}}
Parse and cache templates at startup, not per request

🧪 Phase 8 — Testing & Tooling

Goal: Write tests like a professional Go developer. Automate quality.

Module 8.1 — Unit Testing

Tests live in _test.go files in the same package (or _test suffix for black-box tests)
Test functions: func TestFuncName(t *testing.T)
t.Error / t.Errorf — marks test as failed, continues
t.Fatal / t.Fatalf — marks failed and stops the test immediately
Table-driven tests — the idiomatic Go test pattern
t.Run("subtest name", func(t *testing.T) { ... }) — subtests
Run specific tests: go test -run TestFuncName ./...
Run with verbose output: go test -v ./...
func TestAdd(t *testing.T) {
    tests := []struct {
        name     string
        a, b     int
        expected int
    }{
        {"positive", 2, 3, 5},
        {"negative", -1, -2, -3},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            if got := Add(tt.a, tt.b); got != tt.expected {
                t.Errorf("Add(%d,%d) = %d, want %d", tt.a, tt.b, got, tt.expected)
            }
        })
    }
}

Module 8.2 — Benchmarks & Coverage

Benchmark functions: func BenchmarkFuncName(b *testing.B) { for i := 0; i < b.N; i++ { ... } }
Run benchmarks: go test -bench=. -benchmem ./...
Read benchmark output: ns/op, B/op, allocs/op
Coverage: go test -coverprofile=c.out ./... then go tool cover -html=c.out
Fuzz testing (Go 1.18+): func FuzzFuncName(f *testing.F)

Module 8.3 — Mocking & Test Doubles

Use interfaces to make code testable — inject dependencies
Write manual fakes / stubs implementing the same interface
testify library: assert.Equal, assert.NoError, require.Equal
mockery or gomock — auto-generate mocks from interfaces
HTTP server mocking: httptest.NewServer and httptest.NewRecorder

Module 8.4 — Go Tooling Ecosystem

golangci-lint — run 50+ linters in one command. Configure with .golangci.yml
staticcheck — advanced static analysis
goimports — like gofmt but also manages imports
air — live reload for development
delve (dlv) — Go debugger, integrates with VS Code
pprof — profiling CPU and memory, visualize with go tool pprof
Write a Makefile with targets: build, test, lint, run
